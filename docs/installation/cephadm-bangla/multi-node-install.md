### 1.2 প্রি-রিকোয়ারমেন্ট চেকলিস্ট (বাস্তব উদাহরণসহ)

| প্রি-রিকোয়ারমেন্ট | টেকনিক্যাল বিবরণ | বাস্তব উদাহরণ | কেন গুরুত্বপূর্ণ |
|-----------------|----------------|-------------|---------------|
| **ন্যূনতম নোড** | প্রডাকশনের জন্য ৩টি নোড | ৩-পায়ার স্টুলের মতো - ১টি প্যা ভাঙলেও দাঁড়িয়ে থাকে। ২টি নোড = ঝুঁকিপূর্ণ (১টি ফেল করলে সিস্টেম বন্ধ)। | হাই-আভেইলেবিলিটি নিশ্চিত করে। ১টি নোড ফেল করলেও বাকি ২টি Quorum বজায় রেখে ক্লাস্টার চালু রাখে। |
| **নেটওয়ার্ক সেপারেশন** | পাবলিক নেটওয়ার্ক + ক্লাস্টার নেটওয়ার্ক | হাসপাতালের মতো: পাবলিক নেটওয়ার্ক = রোগীদের প্রবেশদ্বার, ক্লাস্টার নেটওয়ার্ক = স্টাফদের জন্য আলাদা করিডোর যেখানে জরুরি সরঞ্জাম চলাচল করে। | রিপ্লিকেশন ট্রাফিক (OSD থেকে OSD) ক্লায়েন্ট রিকোয়েস্টের গতি কমাবে না। নেটওয়ার্ক কনজেশন রোধ করে। |
| **টাইম সিঙ্ক (NTP/Chrony)** | সব নোডের সময় একই হতে হবে (১০০মি-সেকেন্ডের মধ্যে) | টিম মিটিংয়ের মতো: সবার ঘড়ি যদি আলাদা সময় দেখায়, তাহলে সমন্বয় করা অসম্ভব। | Ceph ডেটা অর্ডারিংয়ের জন্য টাইমস্ট্যাম্প ব্যবহার করে। ১০০মি-সেকেন্ডের বেশি পার্থক্য Quorum ব্রেক করে এবং ডেটা অসামঞ্জস্যতা তৈরি করে। |
| **DNS/হোস্ট রেজোলিউশন** | প্রতিটি নোড অন্য নোডের হোস্টনেম রেজলভ করতে পারবে | ফোন ডিরেক্টরির মতো: কলিগদের কল করতে নামের সাথে নম্বর মিলিয়ে নিতে হয়। | cephadm হোস্টনেম ব্যবহার করে নোড ম্যানেজ করে। ভুল রেজোলিউশন = ডিপ্লয়মেন্ট ফেল। |
| **ডিস্ক প্রিপারেশন** | OSD ডিস্ক অবশ্যই খালি হতে হবে (কোনো পার্টিশন বা ফাইলসিস্টেম থাকবে না) | নতুন নোটবুকের মতো: পুরনো লেখা ভরা পাতায় নতুন নোট লেখা যায় না। | cephadm বিদ্যমান ডেটা থাকা ডিস্কে OSD তৈরি করতে পারবে না। ক্লিন করার জন্য `wipefs -a /dev/sdX` ব্যবহার করুন। |
| **কন্টেইনার রানটাইম** | Docker CE v20.10+ অথবা Podman v3.4+ | শিপিং কন্টেইনার পোর্টের মতো: cephadm সার্ভিস ডিপ্লয় করার জন্য স্ট্যান্ডার্ড কন্টেইনার (Docker) প্রয়োজন। | cephadm কন্টেইনারের ভেতরে Ceph সার্ভিস চালায়। রানটাইম না থাকলে = Ceph চলবে না। |
| **ফায়ারওয়াল রুলস** | প্রয়োজনীয় পোর্ট ওপেন রাখুন: 6789, 3300, 6800-7300, 8443 | বিল্ডিং সিকিউরিটির মতো: শুধুমাত্র নির্দিষ্ট দরজা (পোর্ট) দিয়ে প্রবেশের অনুমতি দিন, বাকিগুলো বন্ধ রাখুন। | ব্লক করা পোর্ট = নোডগুলো একে অপরের সাথে কথা বলতে পারবে না = ক্লাস্টার তৈরি বা রিপ্লিকেশন ফেল। |
| **কার্নেল প্যারামিটার** | `vm.swappiness=1`, `net.ipv4.ip_forward=1` | গাড়ির ইঞ্জিন টিউনিংয়ের মতো: ডিফল্ট সেটিংস কাজ করে, কিন্তু অপ্টিমাইজড সেটিংস ভালো পারফরমেন্স দেয়। | মেমোরি সোয়াপিং রোধ করে (যা Ceph-কে স্লো করে) এবং সঠিক নেটওয়ার্ক রাউটিং এনেবল করে। |

### 1.3 নেটওয়ার্ক আর্কিটেকচার (প্রডাকশন রিকমেন্ডেড)

```
┌─────────────────────────────────────────┐
│           ফিজিক্যাল নেটওয়ার্ক লেআউট      │

│                                          │
│  ┌─────────────┐     ┌─────────────┐    │

│  │        Ceph Node (প্রতিটি)        │    │

```

## 2. মূল কম্পোনেন্ট ও টার্মিনোলজি (বাস্তব উদাহরণসহ)

### 2.1 এসেনশিয়াল কম্পোনেন্ট টেবিল

| কম্পোনেন্ট | কাজ কী করে | বাস্তব উদাহরণ | প্রডাকশন ইমপ্যাক্ট |
|-----------|-----------|-------------|------------------|
| **Monitor (MON)** | ক্লাস্টার স্টেট ম্যাপ মেইনটেইন করে, অথেন্টিকেশন হ্যান্ডেল করে | লাইব্রেরি ক্যাটালগ সিস্টেমের মতো: এটি বই (ডেটা) স্টোর করে না, কিন্তু জানে প্রতিটি বই কোথায় আছে। | প্রডাকশনের জন্য ন্যূনতম ৩টি MON প্রয়োজন। MON গুলো Quorum হারালে ক্লাস্টার Read-Only মোডে চলে যাবে। |
| **Manager (MGR)** | মেট্রিক্স সংগ্রহ করে, ড্যাশবোর্ড চালায়, প্লাগইন হ্যান্ডেল করে | বিল্ডিং ম্যানেজারের মতো: বিদ্যুৎ, পানি, সিকিউরিটি ক্যামেরা মনিটর করে এবং মালিকদের রিপোর্ট দেয়। | Grafana/Prometheus এর মাধ্যমে মনিটরিং এনেবল করে। GUI ম্যানেজমেন্টের জন্য ড্যাশবোর্ড দেয়। |
| **OSD (Object Storage Daemon)** | ডিস্কে ডেটা স্টোর করে, রিপ্লিকেশন ও রিকভারি হ্যান্ডেল করে | গুদাম কর্মীর মতো: আইটেম গ্রহণ করে, তাকে তাকে (ডিস্কে) রাখে, এবং হারানো আইটেম খুঁজে/মেরামত করতে সাহায্য করে। | প্রতিটি ডিস্কের জন্য একটি OSD। ডিস্ক ফেল করলে শুধু সেই OSD ডাউন হয়। ক্লাস্টার অন্য কপি থেকে অটো রিকভার করে। |
| **Metadata Server (MDS)** | শুধুমাত্র CephFS (ফাইল স্টোরেজ) এর জন্য প্রয়োজন, RBD/RGW এর জন্য নয় | ফাইল ক্যাবিনেট ইনডেক্সের মতো: লোকেশন দিয়ে নয়, নাম দিয়ে ফাইল খুঁজতে সাহায্য করে। | ব্লক/অবজেক্ট স্টোরেজের জন্য প্রয়োজন নেই। শুধুমাত্র CephFS ব্যবহার করলে ডিপ্লয় করুন। |

### 2.2 ক্রিটিক্যাল টার্মিনোলজি (সমস্যা ও সমাধানসহ)

#### 🔹 Quorum (কোরাম)

**সংজ্ঞা:** ক্লাস্টার স্টেট নিয়ে সিদ্ধান্ত নেওয়ার জন্য মনিটর নোডগুলোর মধ্যে প্রয়োজনীয় সংখ্যাগরিষ্ঠতা। সূত্র: `(N/2) + 1` যেখানে N = মোট MON সংখ্যা।

**বাস্তব উদাহরণ:** 
> ধরুন ৫ জন পরিচালকের একটি বোর্ড সিদ্ধান্ত নিচ্ছে। বৈধ সিদ্ধান্ত নিতে কমপক্ষে ৩ জন সদস্য (৫০% + ১) উপস্থিত থাকতে হবে। যদি মাত্র ২ জন আসেন, তাহলে কোনো অফিশিয়াল সিদ্ধান্ত নেওয়া যাবে না।

**সাধারণ সমস্যা:** 
> "ক্লাস্টার `HEALTH_WARN: mon clock skew detected` অথবা `mon map epoch mismatch` এ আটকে আছে"

**মূল কারণ:** 
> - নোডগুলোর মধ্যে টাইম সিঙ্ক্রোনাইজেশন ফেলুর
> - নেটওয়ার্ক পার্টিশনের কারণে কিছু MON বিচ্ছিন্ন হয়ে যাওয়া
> - একাধিক নোডে MON প্রসেস ক্র্যাশ করা

**সমাধান:**
```bash
# ১. সব নোডে টাইম সিঙ্ক চেক করুন
chronyc sources -v

# ২. MON স্ট্যাটাস চেক করুন
cephadm shell -- ceph mon dump

# ৩. যদি একটি MON ডাউন থাকে, সেটি রিস্টার্ট করুন
cephadm shell -- ceph orch restart mon.ceph-node2

# ৪. যদি একাধিক MON ডাউন থাকে, ম্যানুয়ালি Quorum রিস্টোর করুন (অ্যাডভান্সড)
# সব MON বন্ধ করুন, একটি --force-quorum ফ্ল্যাগ সহ চালু করুন, তারপর অন্যগুলো যুক্ত করুন
```

**প্রতিরোধ:** 
- সবসময় বিজোড় সংখ্যক MON ব্যবহার করুন (৩, ৫, ৭)
- আলাদা ফিজিক্যাল হোস্টে MON ডিপ্লয় করুন
- সব নোডে নির্ভরযোগ্য NTP/Chrony নিশ্চিত করুন

#### 🔹 CRUSH Map (ক্রাশ ম্যাপ)

**সংজ্ঞা:** অ্যালগরিদম যা ক্যালকুলেট করে ডেটা কোথায় স্টোর হবে, কোনো সেন্ট্রাল লুকআপ টেবিল ছাড়াই। ক্লায়েন্ট সরাসরি লোকেশন কম্পিউট করে।

**বাস্তব উদাহরণ:**
> স্মার্ট পার্কিং সিস্টেমের মতো: গার্ডকে জিজ্ঞেস না করে "আমি কোথায় পার্ক করব?", আপনার গাড়ির GPS রুলস অনুযায়ী সেরা স্পট বের করে দেয়: "ইলেকট্রিক হলে Zone A, SUV হলে Zone B, রাশ আওয়ারে এক্সিটের কাছে এড়িয়ে চলুন।"

**সাধারণ সমস্যা:**
> "ডেটা আনব্যালেন্সড: কিছু OSD ৯০% ফুল, অন্যগুলো ৩০% ফুল"

**মূল কারণ:**
> - CRUSH রুলস ফিজিক্যাল লেআউটের সাথে মেলেনি
> - পুল সাইজের জন্য ভুল `pg_num` ক্যালকুলেশন
> - নতুন OSD যোগ করার পর রি-ব্যালেন্স না করা

**সমাধান:**
```bash
# ১. ডেটা ডিস্ট্রিবিউশন চেক করুন
cephadm shell -- ceph osd df

# ২. বর্তমান CRUSH ম্যাপ দেখুন
cephadm shell -- ceph osd crush dump

# ৩. ক্লাস্টার রি-ব্যালেন্স করুন (অটোমেটিক, কিন্তু ম্যানুয়ালি ট্রিগার করা যায়)
cephadm shell -- ceph osd reweight-by-utilization

# ৪. নতুন পুলের জন্য সঠিক pg_num ক্যালকুলেট করুন:
# সূত্র: (OSD_count * 100) / replication_factor
# উদাহরণ: ৬টি OSD, রিপ্লিকেশন ৩ → (৬*১০০)/৩ = ২০০ → ২৫৬ ব্যবহার করুন (২-এর পাওয়ার)
cephadm shell -- ceph osd pool create mypool 256 256
```

**প্রতিরোধ:**
- ডিপ্লয়মেন্টের আগে CRUSH হায়ারার্কি প্ল্যান করুন (host → rack → datacenter)
- সঠিক মানসহ `cephadm shell -- ceph osd pool create <name> <pg_num> <pgp_num>` ব্যবহার করুন
- নিয়মিত ইউটিলাইজেশন মনিটর করুন

#### 🔹 Split Brain (স্প্লিট ব্রেইন)

**সংজ্ঞা:** নেটওয়ার্ক ফেলুরের কারণে ক্লাস্টার দুটি গ্রুপে বিভক্ত হয়ে যায়, যার প্রতিটি নিজেকে "আসল" ক্লাস্টার মনে করে। উভয়ই রাইট এক্সেপ্ট করতে পারে, ফলে ডেটা কনফ্লিক্ট তৈরি হয়।

**বাস্তব উদাহরণ:**
> দুটি ব্যাংক ব্রাঞ্চের মধ্যে কানেকশন হারিয়ে গেল। উভয়ই ট্রানজেকশন প্রসেস করা চালিয়ে যায়। কানেকশন ফিরে আসার পর, একই অ্যাকাউন্টে দুটি ভিন্ন ব্যালেন্স দেখা যায়। কোনটি সঠিক?

**সাধারণ সমস্যা:**
> "ক্লাস্টার দেখাচ্ছে `HEALTH_ERR: split brain detected` অথবা নেটওয়ার্ক রিকভারির পর ডেটা ইনকনসিসটেন্সি"

**মূল কারণ:**
> - MON নোডগুলোর মধ্যে নেটওয়ার্ক পার্টিশন
> - অপর্যাপ্ত Quorum (যেমন: ৩-নোড ক্লাস্টারে ২টি MON, উভয়ই আইসোলেটেড)
> - ভুল ফায়ারওয়াল কনফিগারেশন যা MON কমিউনিকেশন ব্লক করে

**সমাধান:**
```bash
# ১. চিহ্নিত করুন কোন পার্টিশনের সংখ্যাগরিষ্ঠতা আছে
cephadm shell -- ceph mon dump

# ২. সংখ্যালঘু পার্টিশনে, ডেটা করাপশন রোধ করতে সব Ceph সার্ভিস বন্ধ করুন
systemctl stop ceph-mon@*.service
systemctl stop ceph-osd@*.service

# ৩. নেটওয়ার্ক কানেক্টিভিটি রিস্টোর করুন

# ৪. সংখ্যাগরিষ্ঠ পার্টিশনে, ক্লাস্টার হেলথ ভেরিফাই করুন
cephadm shell -- ceph -s

# ৫. যদি ডেটা কনফ্লিক্ট হয়ে থাকে, ম্যানুয়াল রিকভারি প্রয়োজন হতে পারে (Ceph সাপোর্টে যোগাযোগ করুন)
```

**প্রতিরোধ:**
- সবসময় বিজোড় সংখ্যক MON বজায় রাখুন
- রিডান্ডেন্ট নেটওয়ার্ক পাথ ব্যবহার করুন (বন্ডিং, মাল্টিপল সুইচ)
- MON কমিউনিকেশনের জন্য সঠিক ফায়ারওয়াল রুলস কনফিগার করুন
- `ping`, `mtr`, `iperf3` টুলস দিয়ে নেটওয়ার্ক হেলথ মনিটর করুন

#### 🔹 Placement Groups (PGs) (প্লেসমেন্ট গ্রুপ)

**সংজ্ঞা:** অবজেক্টগুলোর লজিক্যাল গ্রুপিং যা OSD ম্যানেজমেন্টকে সহজ করে। মেটাডেটা ওভারহেড কমায়।

**বাস্তব উদাহরণ:**
> লাইব্রেরিতে বই সাজানোর মতো: ১০,০০০ আলাদা বই ট্র্যাক না করে, আপনি ১০০টি "সেকশন" (PG) ট্র্যাক করেন। প্রতিটি সেকশনে ~১০০টি বই থাকে। বই খুঁজতে মানে তার সেকশন খুঁজে বের করা।

**সাধারণ সমস্যা:**
> "PG গুলো `incomplete` অথবা `down` স্টেটে আছে" অথবা "Too few PGs per OSD" ওয়ার্নিং

**মূল কারণ:**
> - পুল ক্রিয়েশনের সময় ভুল `pg_num`
> - OSD ফেলুরের কারণে PG গুলো প্রয়োজনীয় কপি হারিয়ে ফেলে
> - ক্লাস্টার সম্পূর্ণ ইনিশিয়ালাইজ না হওয়া

**সমাধান:**
```bash
# ১. PG স্ট্যাটাস চেক করুন
cephadm shell -- ceph pg stat
cephadm shell -- ceph pg dump_stuck

# ২. আটকে থাকা PG এর জন্য, রিকভারি ট্রাই করুন
cephadm shell -- ceph pg repair <pg_id>

# ৩. যদি পুলে খুব কম PG থাকে, বাড়ান (সতর্কতা: ডেটা মাইগ্রেশন প্রয়োজন)
cephadm shell -- ceph osd pool set mypool pg_num 512
cephadm shell -- ceph osd pool set mypool pgp_num 512

# ৪. রি-ব্যালেন্সিং সম্পন্ন হওয়া পর্যন্ত অপেক্ষা করুন
cephadm shell -- ceph -s  # "active+clean" না আসা পর্যন্ত চেক করুন
```

**প্রতিরোধ:**
- পুল ক্রিয়েট করার আগে PG ক্যালকুলেট করুন: [Ceph PG Calculator](https://www.ceph.com/pgcalc/)
- ছোট ক্লাস্টারের জন্য `pg_num = 32` দিয়ে শুরু করুন, প্রয়োজন অনুযায়ী বাড়ান
- PG ওয়ার্নিংয়ের জন্য `ceph health detail` মনিটর করুন

#### 🔹 Journal/WAL (রাইট-আহেড লগ)

**সংজ্ঞা:** দ্রুত স্টোরেজ (SSD/NVMe) যা সাময়িকভাবে রাইট স্টোর করে ধীর HDD-তে ফ্লাশ করার আগে। রাইট পারফরমেন্স উন্নত করে।

**বাস্তব উদাহরণ:**
> রেস্তোরাঁ অর্ডার সিস্টেমের মতো: ওয়েটাররা ছোট নোটপ্যাডে (WAL) তাৎক্ষণিক অর্ডার লিখে নেয়, তারপর কিচেন স্টাফ পরে মূল অর্ডার বুকে (HDD) ট্রান্সফার করে। কাস্টমাররা দ্রুত কনফার্মেশন পায়।

**সাধারণ সমস্যা:**
> "HDD-ভিত্তিক OSD-তে স্লো রাইট পারফরমেন্স" অথবা "WAL ডিভাইস ফুল"

**মূল কারণ:**
> - WAL-এর জন্য আলাদা SSD ছাড়া HDD ব্যবহার করা
> - রাইট ওয়ার্কলোডের জন্য WAL ডিভাইস খুব ছোট
> - WAL ডিভাইস ফেলুর

**সমাধান:**
```bash
# ১. OSD পারফরমেন্স চেক করুন
cephadm shell -- ceph osd perf

# ২. যদি HDD ব্যবহার করেন, OSD ক্রিয়েশনের সময় WAL/DB-এর জন্য আলাদা SSD যুক্ত করুন:
cephadm shell -- ceph orch daemon add osd ceph-node1:/dev/sdb:/dev/nvme0n1p1
# ফরম্যাট: <data_disk>:<wal_db_device>

# ৩. বিদ্যমান OSD-এর জন্য, মাইগ্রেশন জটিল - নতুন OSD দিয়ে রিপ্লেস করার কথা বিবেচনা করুন

# ৪. WAL ইউসেজ মনিটর করুন
cephadm shell -- ceph daemon osd.<id> bluestore allocator stats block
```

**প্রতিরোধ:**
- HDD OSD-এর জন্য: WAL/DB-এর জন্য সবসময় SSD পেয়ার করুন (রেশিও ~১:১০, যেমন ১TB HDD-এর জন্য ১০০GB SSD)
- NVMe OSD-এর জন্য: আলাদা WAL প্রয়োজন নেই
- `smartctl` দিয়ে ডিস্ক হেলথ মনিটর করুন

## 3. VM/হার্ডওয়্যার স্পেসিফিকেশন (প্রতিটি নোডের জন্য)

| রিসোর্স | মিনিমাম (ল্যাব) | প্রডাকশন (এন্টারপ্রাইজ) | নোটস |
|--------|---------------|----------------------|-------|
| **CPU** | 4 কোর | 8+ কোর (Xeon/EPYC) | এনক্রিপশন, কম্প্রেশন এবং প্যারালাল OSD অপারেশনের জন্য বেশি কোর সাহায্য করে |
| **RAM** | 8 GB | 16+ GB (প্রতি OSD-এর জন্য 1GB + OS-এর জন্য 4GB) | OSD গুলো ক্যাশিংয়ের জন্য RAM ব্যবহার করে। অপর্যাপ্ত RAM = স্লো পারফরমেন্স |
| **OS Disk** | 50 GB SSD | 100 GB NVMe (RAID 1) | Ubuntu + Docker + Ceph কন্টেইনারের জন্য। রেডান্ডেন্সির জন্য RAID 1 ভালো |
| **Data Disk (OSD)** | 1x 100 GB | 4x 4TB HDD (RAID ছাড়া) + WAL-এর জন্য 1x 480GB SSD | Ceph রেডান্ডেন্সি হ্যান্ডেল করে। OSD-এর জন্য হার্ডওয়্যার RAID ব্যবহার করবেন না |
| **Network** | 1x 1 GbE | 2x 10 GbE (আলাদা) | পাবলিক: ক্লায়েন্ট ট্রাফিক। ক্লাস্টার: রিপ্লিকেশন ট্রাফিক। জাম্বো ফ্রেম (MTU 9000) রিকমেন্ডেড |
| **OS** | Ubuntu 24.04.4 LTS | Ubuntu 24.04.4 LTS (লেটেস্ট কার্নেল) | সব নোডে একই OS ভার্সন রাখুন |
| **Swap** | ডিজেবল | ডিজেবল (`vm.swappiness=1`) | সোয়াপিং Ceph-কে স্লো করে। `sudo swapoff -a` ব্যবহার করুন এবং `/etc/fstab` থেকে রিমুভ করুন |

## 4. ধাপে-ধাপে ইনস্টলেশন গাইড (A-to-Z)

```bash
# ১. সিস্টেম আপডেট করুন
sudo apt update && sudo apt upgrade -y
```

### Docker ইনস্টলেশন (সব নোডে রান করুন)

```bash
# ১. পুরনো/কনফ্লিক্টিং Docker প্যাকেজ রিমুভ করুন
for pkg in docker.io docker-doc docker-compose docker-compose-v2 podman-docker containerd runc; do 
    sudo apt remove -y $pkg
done

# ২. Docker-এর অফিশিয়াল GPG Key অ্যাড করুন
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# ৩. Docker রিপোজিটরি অ্যাড করুন
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# ৪. Docker ইনস্টল করুন
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

# ৫. Docker স্টার্ট ও এনেবল করুন
sudo systemctl enable --now docker

# ৬. বর্তমান ইউজারকে docker গ্রুপে অ্যাড করুন (docker কমান্ডের জন্য sudo এড়ানোর জন্য)
sudo usermod -aG docker $USER
newgrp docker  # লগআউট ছাড়া গ্রুপ চেঞ্জ অ্যাপ্লাই করুন

# ৭. Docker ভেরিফাই করুন
docker --version
docker run hello-world  # "Hello from Docker!" মেসেজ দেখাবে
```

### সাধারণ সমস্যা ও সমাধান

| সমস্যা | লক্ষণ | সমাধান |
|-------|-------|--------|
| **OSD ডাউন** | `ceph osd tree` এ OSD কে "down" দেখায় | ১. ডিস্ক হেলথ চেক করুন: `smartctl -a /dev/sdb`<br>২. OSD রিস্টার্ট করুন: `cephadm shell -- ceph orch restart osd.<id>`<br>৩. যদি ডিস্ক ফেল করে: হার্ডওয়্যার রিপ্লেস করুন, তারপর `ceph orch apply osd --all-available-devices` |
| **স্লো রাইট** | হাই লেটেন্সি, লো থ্রুপুট | ১. নেটওয়ার্ক চেক করুন: `iperf3 -c <node>`<br>২. HDD OSD-এর জন্য WAL SSD-তে আছে কিনা ভেরিফাই করুন<br>৩. রি-ব্যালেন্সিং চলছে কিনা চেক করুন: `ceph -s` (রি-ব্যালেন্সিং চলাকালীন হেভি রাইট এড়িয়ে চলুন) |
| **MON ফ্ল্যাপিং** | MON বারবার up/down হচ্ছে | ১. টাইম সিঙ্ক চেক করুন: `chronyc sources`<br>২. MON নোডে ডিস্ক স্পেস চেক করুন<br>৩. লগ রিভিউ করুন: `cephadm shell -- ceph logs mon.ceph-node1` |
| **PG আটকে যাওয়া** | `ceph pg dump_stuck` এ এন্ট্রি দেখায় | ১. রিপেয়ার ট্রাই করুন: `ceph pg repair <pg_id>`<br>২. যদি স্থায়ী হয়, OSD হেলথ চেক করুন<br>৩. শেষ উপায় হিসেবে: `ceph pg force_create_pg <pg_id>` (সতর্কতার সাথে ব্যবহার করুন) |
| **ড্যাশবোর্ড লোড না হওয়া** | ব্রাউজারে কানেকশন এরর দেখায় | ১. সার্ভিস চেক করুন: `cephadm shell -- ceph mgr services`<br>২. ফায়ারওয়াল ভেরিফাই করুন: `sudo ufw status`<br>৩. সার্টিফিকেট চেক করুন: `openssl s_client -connect <ip>:8443` |

## ফাইনাল ভেরিফিকেশন

সব ধাপ সম্পন্ন করার পর, এই ফাইনাল চেক রান করুন:

```bash
sudo cephadm shell -- ceph -s
```

✅ **হেলদি ক্লাস্টারের জন্য প্রত্যাশিত আউটপুট:**
```
  cluster:
    id:     xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    health: HEALTH_OK
 
  services:
    mon: 3 daemons, quorum ceph-node1,ceph-node2,ceph-node3
    mgr: ceph-node1.active
    osd: 3 osds: 3 up, 3 in
 
  
    pools:   1 pools, 128 pgs
    objects: 0 objects, 0 B
    usage:   0 B used, 0 B / 0 B avail
    pgs:     128 active+clean
```

🎉 **অভিনন্দন!** আপনি এখন Ubuntu 24.04.4-এ cephadm ব্যবহার করে একটি প্রডাকশন গ্রেড, এন্টারপ্রাইজ লেভেল Ceph Cluster চালু করেছেন।

---

> **সাহায্য প্রয়োজন?**  
> - অফিশিয়াল ডকস: https://docs.ceph.com/en/latest/  
> - কমিউনিটি: https://ceph.io/community/  
> - এন্টারপ্রাইজ সাপোর্টের জন্য: Red Hat অথবা SUSE-এর সাথে যোগাযোগ করুন (Ceph-এর এন্টারপ্রাইজ ডিস্ট্রিবিউটর)

এই গাইডটি একটি শক্তিশালী ফাউন্ডেশন প্রদান করে। আপনার ক্লাস্টার বড় হওয়ার সাথে সাথে, Erasure Coding, মাল্টি-সাইট রিপ্লিকেশন, এবং Kubernetes-এর সাথে ইন্টিগ্রেশন (Rook/Ceph)-এর মতো অ্যাডভান্সড টপিক এক্সপ্লোর করুন। প্রডাকশনে অ্যাপ্লাই করার আগে সবসময় ল্যাব এনভায়রনমেন্টে চেঞ্জ টেস্ট করুন।
